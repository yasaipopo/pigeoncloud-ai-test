## STEP 1. 機能の洗い出し

対象画面のソースコード（Angular HTML + TypeScript）を読み、全機能を一覧化する。

- [ ] HTMLテンプレートから全ボタン・メニュー項目・操作可能UI要素を列挙
- [ ] TypeScriptから全publicメソッド（ユーザー操作に対応するもの）を列挙
- [ ] 機能を以下の分類で整理：
  - **CRUD操作**: 作成・表示・編集・削除があるもの
  - **状態変更**: 設定ON/OFF、並び替え、移動
  - **表示系**: 初期表示、条件付き表示、権限による表示制御
- [ ] 各機能に対して「テストすべきか」「環境制約で不可か」を判断

## STEP 2. テストケースの設計（グループ化）

機能一覧から、テストケース（dash-010等）とmovie（DB01等）を設計する。

- [ ] 関連する機能をグループ化してmovieにまとめる（1 movie = 1動画 = 1テスト関数）
- [ ] 各movieのケース構成を決める。以下を考慮：
  - **CRUD全操作を1つのフローで通す**: 作成→確認→編集→確認→削除→確認
  - **複合パターン**: 複数アイテムを作成して操作（1個だけでなく2〜3個）
  - **同種機能の複数タイプ**: テーブル/集計/チャート/掲示板をそれぞれ追加
  - **操作→リロード→状態保持**: 操作後にページリロードしても結果が残っていること
- [ ] 各ケースに明確なタイトルを付ける（「〜を追加できること」「〜が削除されること」等）
- [ ] ケースのタイトルに対して、結果を直接確認する✅が設計段階で含まれていること
  - 「ビューを追加できること」→ 追加後にビューが表示されている✅が必須
  - 「エラーなし」だけでは不十分

## STEP 3. 共通チェックの定義

ページに戻ってきたときに毎回実行する「ページ正常表示チェック」を定義する。

- [ ] **各画面ごとに共通チェック関数を設計する**
  - ダッシュボード: ナビゲーションメニュー表示、アクティブタブ表示、タブ内コンテンツ表示、エラーなし
  - テーブル管理: ナビゲーション表示、テーブル一覧表示、エラーなし
  - レコード一覧: テーブルヘッダー表示、データ行存在、エラーなし
- [ ] **初回チェックと2回目以降のチェックを分ける**
  - 初回: 各要素を個別に詳しくチェック（タブ名、掲示板のテキスト内容、サイドバー項目等）
  - 2回目以降: 共通チェック関数を呼ぶだけ（「ダッシュボードが正常に表示されていること」）
- [ ] **共通チェック関数はspec.jsにヘルパーとして実装する**
  ```
  例: async function checkDashboardPage(page, options) {
      // ナビゲーション表示
      // アクティブタブ表示  
      // タブコンテンツ表示（スケルトンでないこと）
      // エラーなし
  }
  ```
- [ ] **操作後のページ遷移・リロード後には必ず共通チェックを呼ぶ**

## STEP 4. 各ケースの詳細ステップ設計

各テストケースの操作手順と確認ポイントを設計する。

### 操作手順（[flow]行）の設計

- [ ] テスター向けの日本語で書く（CSSセレクター・Playwrightコード・URLパス禁止）
- [ ] ボタン・メニュー名は「」で囲む
- [ ] 画面名は「○○画面」と書く
- [ ] 「モーダル」ではなく「ダイアログ」
- [ ] 番号はケース内番号（30-1, 30-2...）

### 確認ポイント（[check]行 / ✅）の設計

- [ ] 各ケースのタイトルの結果を直接確認する✅が最低1つある
- [ ] 「エラーがないこと」を確認する場合、エラーメッセージ非表示 AND Internal Server Error非含有の両方
- [ ] 「〜が表示されていること」の確認では、要素の存在だけでなく中身も確認する
  - テーブル: ヘッダー + データ行が1行以上（スケルトンでない）
  - ウィジェット: 実データが読み込まれた状態
  - 掲示板: テキストコンテンツが表示されている
- [ ] 「〜が追加/削除されていること」の確認では、追加前後の差分 or 特定要素の有無
- [ ] 「〜が存在しないこと」の確認では、対象を実際に探した上で0件確認
  - 「メニューに削除がないこと」→ メニューを開いた状態で確認
- [ ] 同じ確認を2回目以降でやる場合は、共通チェック関数に委任してよい

### データ表示完了の待ち方

- [ ] ページ遷移後に表示を期待する具体的な要素が出るまで待つ
  - テーブルデータ: `tbody tr td` がスケルトンでなく実データで出るまで
  - ウィジェット: ローディングプレースホルダーが消えるまで
  - 成功トースト: 操作成功時に緑の通知が出るまで
- [ ] 「データはありません」「スケルトン行」の状態でアサーション/スクショを撮らない
- [ ] `waitForTimeout`ではなく`waitForSelector`/`waitForLoadState`/`waitForFunction`を使う

## STEP 5. spec.js実装

### コメント記法

- [ ] 全操作に `// [flow] XX-N. 操作内容` コメント
- [ ] 全確認ポイントに `// [check] XX-N. ✅ 確認内容` コメント
- [ ] 未実施の確認に `// [check] XX-N. 🔴 確認内容（理由）` コメント

### エラーハンドリング

- [ ] 操作対象のUI要素は`expect().toBeVisible()`で確認してからクリック
- [ ] `if (visible)` で操作をスキップしている箇所がない → `expect`に変更
- [ ] `.catch(() => {})` が本当に握りつぶしてよい箇所のみに限定されている
  - OK: チュートリアルダイアログ閉じ（出ない場合もある）
  - NG: テスト対象の操作（ボタンクリック、フォーム入力）

### スクショ

- [ ] autoScreenshotの呼び出しが、そのケースの最後のアサーション完了後にある
- [ ] スクショにローディング状態が写っていない
- [ ] 「〜が存在しないこと」の確認では、対象を探した状態（メニュー開いた状態等）でスクショ

### コード品質

- [ ] コメント番号がyamlのケース内番号と一致（`// [flow] 30-1.`形式）
- [ ] `waitForTimeout`使用は最小限で理由が明確
- [ ] `console.log`にケース番号含む
