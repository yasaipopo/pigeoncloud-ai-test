## データセット管理 / Dataset Management (2026-04-18)

`tests/data-operations.spec.js` の強化に伴う調査・実装メモ。

### 調査要・不明確な項目

1. **楽観ロック (Optimistic Locking)**
   - **状況**: `data-730` で実装。フロントエンドまたは API で `updated_at` 等を用いた競合検知が行われているか不明。
   - **懸念**: 排他制御が実装されていない場合、後勝ちで上書きされ、コンフリクトエラーが出ないためテストが fail する。

2. **下書き保存 (Draft Saving)**
   - **状況**: `data-740` で実装。PigeonCloud に標準で「下書き」機能が存在するか不明。
   - **対応**: ボタンテキストに `下書き`, `一時保存` を指定しているが、UIが存在しない場合はボタンが見つからずスキップに近い挙動になる（`count() > 0` でガード）。

3. **一般ユーザー権限 (Permission Violation)**
   - **状況**: `data-750` で `test_general@example.com` でのログインを想定。
   - **対応**: 環境に当該ユーザーが存在しない場合、ログインに失敗してテストが fail する。`createTestEnv` で作成されるユーザー情報の利用が望ましい。

### 予想される failure (プロダクト未実装疑い)

- **data-730 (楽観ロック)**: プロダクト側に競合検知ロジックがない場合。
- **data-740 (下書き保存)**: 「下書き」という概念自体がプロダクトにない場合。
- **data-760 (バリデーション - 文字数制限)**: フロントエンドで `maxlength` 等が設定されておらず、かつサーバー側でもエラーを返さず切り捨てて保存される場合。

## 決済システム / Payment System (2026-04-18)

`tests/payment.spec.js` の強化に伴う調査・実装メモ。

### 調査要・不明確な項目

1. **返金機能 (Refund)**
   - **状況**: `pay-170` で実装。ユーザー画面（管理画面）から直接の「返金」ボタンやAPIが存在するか不明。
   - **懸念**: 一般的なSaaSでは返金は運営側（Stripeダッシュボード等）で行うことが多いため、プロダクトUIには存在しない可能性がある。

2. **3Dセキュア認証 (3D Secure)**
   - **状況**: `pay-150` で実装。テスト環境で3Dセキュアをトリガーするカード情報が利用可能か不明。
   - **対応**: Stripeのテストカード（3Dセキュア用）を使用して、iframeが展開されるかの確認に留めている。

3. **解約フロー (Cancellation Flow)**
   - **状況**: `pay-110` で実装。UI上に「解約」ボタンが明示的に存在するか、または「お問い合わせ」ベースなのか不明。

### 予想される failure (プロダクト未実装疑い)

- **pay-040 (Stripe入力)**: テスト環境（ai-test.pigeon-demo.com）のStripe APIキーが未設定、または無効な場合にフォームが表示されない。
- **pay-100 (日割り計算表示)**: プラン変更時の画面上に「日割り計算額」が明示的に表示されるロジックが未実装の場合。
- **pay-170 (返金)**: ユーザー向けの返金UI自体が存在しない場合。
- **pay-130 (カード削除)**: 単一のカードしか登録できない仕様で、かつ「削除」ボタンが提供されていない場合。
