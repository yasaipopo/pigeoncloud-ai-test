# 認証テスト実装ノート (2026-04-18)

Step 3-a における tests/auth.spec.js の強化に伴う調査・実装メモ。

## 調査要・不明確な項目

### 1. 2FA バックアップコード (Backup codes / Recovery codes)
- **状況**: `docs/security/authentication.md` の設定例には `recovery_codes: 8` の記述があるが、DBスキーマ (`admin_2fa_secret` テーブル) および `LoginController.php` / `Qr2faService.php` のロジック内にバックアップコードを生成・検証する実装が見当たらない。
- **対応**: テストコードでは項目として残しつつ、自動テストの実装は見送った。プロダクト側で未実装の可能性が高い。

### 2. アカウントロックアウトの解除条件
- **状況**: `LoginController.php` では `created >= date("Y-m-d 00:00:00")` の `login_fail` ログをカウントしている。
- **挙動**: 一度ロックされると、日付が変わるまで（または管理者が手動でログを削除/フラグ更新するまで）ログイン不可となる仕様に見える。「N分後に自動解除」というロジックは標準のログインフロー内には見当たらない。
- **対応**: テストでは「本日はアカウントロックされました」というメッセージの表示を確認するに留めた。

### 3. TOTP コードの生成 (auth-130)
- **状況**: E2Eテスト環境（Playwright）に `otplib` 等の TOTP 生成ライブラリが含まれていないため、シークレットキーから動的にコードを生成して認証成功させるテストの自動化が困難。
- **対応**: 手順（flow）の記述のみ行い、テスト自体は「認証失敗（誤コード）」の確認に重点を置いた。

## 予想される failure (プロダクト未実装疑い)

- **auth-190 (閲覧ユーザーによる API 更新拒否)**: プロダクト側の API エンドポイント（`/api/admin/edit/...`）で、マスター以外のユーザーに対する適切な権限チェックが行われていない場合、403 が返らずにテストが fail する可能性がある。
- **auth-200 (API への直接 POST バイパス)**: 上記と同様。
