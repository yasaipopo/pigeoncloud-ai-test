# テストカバレッジ方針 (PHPUnit と E2E の位置付け)

**方針改訂: 2026-04-20**

## 🎯 基本方針: 両方で全パターンを網羅する

### 旧方針 (廃止) ❌
「PHPUnit でロジック検証済みだから E2E では重複を避ける」
→ 不採用。

### 新方針 ✅
**PHPUnit と E2E の両方で、同じ機能について全パターンを網羅する**

理由:
- PHPUnit = ロジックの正当性を高速検証 (単体/境界値/全組み合わせ)
- E2E = UI 入力 → フォーム → API → DB → ロジック → 表示 の統合経路を実際のブラウザ挙動で検証
- **同じロジックでも経路が違えば検証価値が異なる**
- UI バリデーション、フォーム状態、ブラウザ挙動は E2E でしか検出できない

---

## 🔧 Integration テスト不足への対応

PHPUnit (Unit / Integration) に抜けがあった場合:

1. **まずプロダクト側の PHPUnit を先に修正** (tmprepo 経由で pigeon_cloud へ PR)
2. **その上で E2E でも同じパターンを全網羅** (UI 統合として独立に検証)

**E2E 側のケース削減はしない**。PHPUnit の有無と E2E ケース数は独立。

---

## 除外条件 (E2E で実施不可)

以下のみ E2E をスキップし、`.claude/test-env-limitations.md` に記録:

1. **外部依存**: SAML IdP 連携実走、ALB mTLS 実走、SSO リダイレクト実走など
2. **プロダクト未実装**: 機能自体が存在しない (DB カラムなし、UI なし、バックエンドなし)
3. **破壊的操作**: staging 環境を壊すリスクがある操作

**それ以外の理由でのスキップ禁止**: 「難しい」「面倒」「PHPUnit で済む」は理由にならない。

## E2E 責務: 常に以下を含める

UI/統合経路の検証:
- ✅ UI 入力が DB に保存されるか (ラウンドトリップ)
- ✅ 保存済みの値が UI に再表示されるか
- ✅ UI バリデーションが発動するか (サーバー側だけでなく)
- ✅ 画面遷移が仕様通りか
- ✅ エラー画面の UI 挙動 (トースト、モーダル、リダイレクト)
- ✅ 同じ機能の全パターン (正常/異常/境界値/組み合わせ)

PHPUnit 責務: ロジック単体の高速検証 (同じパターンでも別の目的で実施)

## 追加: プロダクト側 PHPUnit の改善タスク (pigeon_cloud 側で別途対応)

現在の PHPUnit 抜け (gemcli 2026-04-19 確認):

| 項目 | 現状 | アクション |
|------|-----|-----------|
| `NetCommon::getIp()` | markTestSkipped | モック化してテスト実装 |
| `Admin::isValidIp()` | 未テスト | システム+個別ホワイトリスト統合テスト |
| 公開フォーム IP制限 | 未テスト | routes/public テスト追加 |
| API IP制限 | 未テスト | public/api/index.php テスト追加 |

これらは tmprepo 経由で pigeon_cloud の PHPUnit 側で対応。E2E 実装とは独立に進める。

## まとめ: 黄金ループ Step [2] での判断

新規 E2E テスト追加時:

1. ✅ PHPUnit 既存テストを gemcli で確認
2. ✅ **Integration 側の不足あり → プロダクト PR で先に追加**
3. ✅ **E2E 側は PHPUnit と独立に、全パターン網羅**
4. ✅ 例外は `.claude/test-env-limitations.md` に記録 (外部依存/未実装/破壊的のみ)
