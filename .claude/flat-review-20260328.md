# PigeonCloud E2Eテスト フラットレビュー結果

**レビュー日**: 2026-03-28
**レビュアー**: 外部レビュアー（Claude）

---

## 1. チェックシートWeb（sheet.html）

### 1-1. [高] パイプライン進捗が初回表示時に全て0表示になる
- ページ読み込み直後、パイプライン進捗セクションの数値が全て「0」と表示される
- テーブルデータが読み込まれた後に正しい値（OK 1625、NG 233 等）になるが、初回のちらつきは混乱を招く
- **対策**: ローディングスピナーまたはスケルトン表示にして、データ取得完了後に数値を表示する

### 1-2. [高] Staging結果・Main結果が全て0/未実施
- Staging結果: 0/2123、Main結果: 0/2123 で全て未実施
- パイプラインの最終2ステップが完全に空のまま。チェックシート上でStagingやMainの結果を表示・記録する仕組みが未稼働
- **対策**: E2Eビューアーの実行結果（run.html）とチェックシートの同期を実装し、「結果同期」ボタンで自動反映されるようにする

### 1-3. [高] case_noの大量重複（720グループ、1,458件）
- 2,123件のデータ行中、720グループのcase_noが重複しており、重複に巻き込まれている行が1,458件（全体の68.7%）
- 例: case_no `218` が4回出現、`216` `224` `226` `230` `231` `233` 等が3回出現
- 複数のspec/yaml間で同じcase_noが使われている（例: uncategorized内の `216` と fields内の `216` は異なるテスト）
- **対策**: case_noにプレフィックスを付けてグローバルにユニークにする（例: `UC-216`、`FD-216`）、またはspec名をキーに含めた複合IDに変更する

### 1-4. [高] テスト内容がURLだけの行が279件（13%）
- `https://loftal.pigeon-cloud.com/admin/dataset__90/view/XXX` のようなURLだけで、何をテストするか不明なケースが279件
- Slack URL（`loftal.slack.com/archives/...`）が手順として記載されているケースが131件
- **これらはテストケースとして自動化不能**
- **対策**: URLだけの行は手順と期待結果を具体化するか、自動テスト対象外として分離する

### 1-5. [高] 期待結果が「想定通り」のみのケースが374件（17.6%）
- 「想定通りの結果となること。」だけでは何を検証すべきか不明
- E2Eテストの assertionを書けないため、実質テスト不能
- **対策**: 全374件について具体的な検証条件（表示される要素、遷移先URL、エラーメッセージの有無等）を記述する

### 1-6. [中] テスト内容・期待結果が両方空の行が222件
- case_no `JSON-01`〜`JSON-04`、`LOCK-01`〜`LOCK-03`、`ORD-01`〜`ORD-02`、`839-1`〜`844-1` 等
- テストケースとして存在するが内容が未記入
- **対策**: 内容を記入するか、削除する

### 1-7. [中] 本番環境URL・認証情報がテスト手順に含まれている
- `pigeon-cloud.com`（本番）や `pigeon-demo.com`（テスト環境）のURL・パスワードがテスト手順に直書きされている行が199件
- 例: `●テスト環境URL https://demo-user-num.pigeon-demo.com ●ID/PW admin 1rxKLot98PUE`
- **対策**: 環境依存の情報はテストケースから分離し、環境変数で管理する

### 1-8. [低] テスト内容・期待結果カラムの文字が途中で切れる
- テーブル表示で長いテキストが `...` で切れるが、マウスオーバー等で全文を見る方法がない（tooltip等の実装が望ましい）

### 1-9. [低] 動画列の表示
- 動画番号（AT01, CC03等）が表示されているが、クリックして動画が見られるわけではない。この列の意図が不明確

## 2. テスト一覧（YAML）の品質

### 2-1. [高] uncategorized系に793件（全体の37.3%）が未分類のまま
- uncategorized: 507件、uncategorized-2: 167件、uncategorized-3: 119件
- このうち538件はfeature（機能名）すら空
- 251件は「各種画面」という曖昧な機能名
- 実際の内容を見ると、以下のような具体的な機能に分類可能:
  - ワークフロー関連: ワークフローステータス、承認、否認
  - フィールド関連: Yes/No、選択肢、ルックアップ
  - テーブル設定: 埋め込みフォーム、公開フォーム、子テーブル
  - ユーザー管理: ログイン制限、2段階認証、パスワード
  - フィルタ・検索: 簡易検索、混合フィルタ
  - CSV/インポート: CSVアップロード・ダウンロード
  - 通知: メール送信、Webhook
  - チャート・カレンダー
- **対策**: 793件を既存のカテゴリ（workflow, fields, users-permissions等）に振り分け、uncategorized系を解消する

### 2-2. [高] descriptionがテスト手順ではなく「機能説明」になっているケース多数
- uncategorized-2/uncategorized-3の「各種画面」カテゴリでは、descriptionとexpectedが完全に同じ文言
  - 例: description=「Yes/Noフィールドでラベルが空白のまま登録するとバリデーションエラーが発生すること」、expected=同じ文
- これは「操作手順」ではなく「何が起きるべきか」の説明であり、テスト手順としては不十分
- **対策**: 操作ステップ（1. ログイン → 2. テーブル管理を開く → 3. ...)を具体的に書く

### 2-3. [高] movie（動画番号）の割り振りが不合理
- uncategorized内の507件中、動画番号が `UC01`〜`UC10` 程度に大雑把に割り振られている
- 1つの動画番号に50件以上のテストが紐づいている状態
- 動画番号は本来「同じ画面・同じ操作フロー」でグループ化すべきだが、機能がバラバラ
- **対策**: 機能ごとに再整理してから、動画番号を再割り振りする

### 2-4. [中] case_noの命名規則が不統一（13パターン）
- `N-N`形式: 905件（例: `1-1`, `102-10`）
- `N`形式: 1,176件（例: `128`, `295`）
- 特殊プレフィックス: `JSON-N`(4件), `DB-N`(6件), `PAY-N`(6件), `LOCK-N`(3件), `ORD-N`(2件), `RPA-N`(6件), `ARC-N`(1件), `AUTOID-N`(2件), `TMPL-N`(6件)
- `N-N'`（シングルクォート付き）: 2件 ← 明らかなデータ不備
- **対策**: `{機能略称}-{連番}` 形式に統一する

### 2-5. [中] Slack URLやissue番号がdescriptionに含まれている
- `#issue819`、`#issue791` のようなissue参照が多数
- これらはテスト手順ではなく、開発タスクのトラッキング情報
- **対策**: issue参照は別フィールド（metadata等）に分離し、descriptionには操作手順のみ記述する

### 2-6. [中] YAMLとspec.jsの件数乖離
- YAML合計: 2,123件（sheet.html上）
- spec.jsのtest合計: 1,279件（specs.html上）
- 差分844件は「YAMLにはあるがspec.jsに実装されていない」テストケース
- **対策**: 未実装ケースを一覧化し、優先度をつけて実装計画を立てる

## 3. E2Eビューアー全体

### 3-1. [低] トップページ（index.html）: 良好
- 実行履歴カードのデザインが見やすく、PASS/FAIL/SKIPの数値が色分けで直感的
- 全体進捗セクションでリアルタイム更新が見える
- 環境情報（Staging、ブランチ、コミット）も適切に表示
- フィルタ機能（FAILあり/成功のみ/実行中）も問題なし
- **改善点**: 第9回（unknownで0/0/0）のような空実行が表示されている。自動削除 or 非表示にすべき

### 3-2. [低] 実行詳細（run.html）: 良好
- テストケースごとのPASS/FAIL/SKIP、実行時間、添付（動画/スクショ）が一覧で見える
- フィルタボタン（すべて/FAILのみ/PASSのみ/SKIPのみ）と検索ボックスが機能
- エラーメッセージの表示も適切

### 3-3. [低] テストケース詳細（case.html）: 良好
- パンくずリスト（実行一覧 → runId → テスト名）がナビゲーションに便利
- エラーメッセージ、テスト動画、スクリーンショット、テストステップが4セクションに分かれて見やすい
- **改善点**: 動画が「動画を再生できません」になっているケースあり。原因不明（ファイル形式/CORS/存在しない等）

### 3-4. [中] テスト一覧（specs.html）: 概ね良好
- 28個のspecファイルが一覧表示、件数と最終更新日時が見える
- `fields-5.spec.js: 0件` のような空specがある
- **改善点**: 0件のspecは目立つように警告表示にする

### 3-5. [低] ナビゲーション
- トップページ → チェックシート / テスト一覧 のリンクがヘッダーにある
- sheet.htmlにはトップへの戻りリンクがある
- 全体的にナビゲーションは問題なし

## 4. テスト構成

### 4-1. [高] uncategorized系の793件は早急に再分類すべき
- 全2,123件中793件（37.3%）が「未分類」は構造的な問題
- uncategorizedが3ファイルに分割されている理由は件数制限のためだが、そもそも分類すべき
- 再分類先の候補:
  - フィールド関連 → fields系に統合
  - ワークフロー関連 → workflow に統合
  - テーブル設定関連 → table-definition に統合
  - ユーザー・権限関連 → users-permissions に統合
  - 通知関連 → notifications に統合
  - CSV関連 → csv-export に統合
  - チャート関連 → chart-calendar に統合
  - レコード操作関連 → records に統合
  - 残りの「画面表示確認」系 → `smoke-test.yaml` 等の新カテゴリ

### 4-2. [中] fields系の5ファイル分割の妥当性
- fields.yaml: 257件 → fields.spec.js: 32件
- fields-2.yaml: 41件 → fields-2.spec.js: 42件
- fields-3.yaml: 145件 → fields-3.spec.js: 145件
- fields-4.yaml: 6件 → fields-4.spec.js: 6件
- fields-5.yaml: 13件 → fields-5.spec.js: 0件（空！）
- 分割基準が不明確。フィールドタイプ別（文字列/数値/日時/選択/ファイル等）に分割するなら命名でわかるようにすべき
- **対策**: `fields-text.yaml`、`fields-number.yaml`、`fields-datetime.yaml` 等、内容がわかる名前にリネーム。fields-5は空なので削除

### 4-3. [中] spec.jsの実装率が60%（1,279/2,123件）
- 844件のテストケースがYAMLにあるがspec.jsに未実装
- 特にuncategorized系の実装率が低い（507件中238件のみ実装）
- **対策**: 自動化可能/不可能を判定し、自動化不可能なケース（外部サービス依存、10万件データ等）は手動テストに分離

### 4-4. [中] chart-calendar、notifications もそれぞれ2ファイルに分割されている
- chart-calendar: 82件 + chart-calendar-2: 36件
- notifications: 111件 + notifications-2: 42件
- 分割基準が不明確（件数制限？）
- **対策**: 1ファイルに統合するか、分割基準（例: chart系/calendar系）を明確にしてリネーム

### 4-5. [低] 28ファイルという数は概ね妥当
- 機能単位でのファイル分割自体は正しいアプローチ
- ただし実質は「15機能 + 5 fields + 2 chart + 2 notifications + 3 uncategorized + 1 dashboard」で、正味15〜20カテゴリに整理可能

## 総括

最大の構造的問題は **「uncategorized 793件」** と **「case_no重複 1,458件」**。この2つが解消されない限り、テストの管理基盤として機能しない。E2Eビューアーの見た目・使い勝手は良好で、sheet.htmlのフィルタ/検索機能も動作する。ただし、中身のデータ品質（URLだけのテスト内容、曖昧な期待結果、機能名の欠如）が全体の信頼性を大きく損なっている。

テスト基盤として成熟させるためには、以下の順で取り組むことを推奨:
1. case_noの重複解消（グローバルユニーク化）
2. uncategorized 793件の機能別再分類
3. URLだけ/空の行の整理（自動テスト対象外に分離）
4. 「想定通り」の期待結果を具体化
5. spec.js未実装ケースの計画的実装
