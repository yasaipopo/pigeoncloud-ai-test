# E2E テストスキップ率分析レポート (2026-04-20)

## 1. エグゼクティブサマリー
現状の priority:high 要件におけるスキップ率は **15.8% (12/76件)** であり、業界標準の「健全圏 (<5%)」を大きく超える「警戒・危機圏」にある。主な要因は「外部 SaaS 依存」と「プロダクト未実装」であり、モック導入と機能実装後の追随により **7.9% までの短期的削減が可能** である。

## 2. スキップ項目の分類 (全体)

`.claude/test-env-limitations.md` に基づき、全スキップ項目を以下のカテゴリに分類した。

| カテゴリ | 内容 | 該当項目例 | 削減可能性 |
| :--- | :--- | :--- | :--- |
| **A. インフラ依存** | AWS 経路, DNS, プライベートIP 等 | SEC_DOMAIN, B002 再現 | 低 (Debug API で回避) |
| **B. 外部 SaaS 依存** | Stripe, SAML IdP, SMTP | SAML 実ログイン, Stripe Webhook | **高 (Sandbox/CLI 活用)** |
| **C. プロダクト未実装** | 機能自体が未だ存在しない | 一時的な許可, 返金機能 | **高 (実装後に即追加)** |
| **D. 破壊的操作** | 環境破壊リスクが高いもの | (現在は特になし) | - |
| **E. Playwright 制限** | ブラウザの証明書インポート等 | 失効証明書アクセス | 中 (API/PHPUnit 併用) |

## 4. 削減可能なスキップ (批判的レビュー)

現状の「技術的に不可避」という判断に対し、以下の 3 点で改善の余地がある。

1.  **外部 SaaS の「実走」への拘りすぎ**:
    *   SAML や Stripe は、外部サービスそのものをテストするのではなく「PigeonCloud が正しいリクエストを送り、正しいレスポンスを処理できるか」をテストすべきである。モック IdP や Stripe CLI を導入すれば、B (外部 SaaS 依存) の 5 件は即座に解消できる。
2.  **Debug API の未活用**:
    *   IP 制限やドメインバイパス等のインフラ依存項目は、プロダクト側に `X-Debug-Client-Ip` ヘッダー等を許可するテスト用フラグを 1 つ追加するだけで E2E 可能になる。「本物」に拘らず「統合」を優先すべき。
3.  **未実装機能の放置**:
    *   C (プロダクト未実装) はテスト側の負債ではないが、スキップ率を押し上げている。これらを「Pending」として別管理し、実装 PR とテスト PR をセットにする運用が必要。

## 5. 業界標準との比較 & 現実的な目標値

*   **業界標準**: 一般的な SaaS ではスキップ率 **5% 未満** が「信頼できるスイート」の基準。10% を超えると「壊れ窓理論」によりテスト結果が無視され始める。
*   **PigeonCloud の現状 (15.8%)**: 「危機的状況」にあり、エンジニアが fail を「また環境のせいか」と見過ごすリスクが高い。
*   **現実的な目標**:
    *   **短期 (1ヶ月)**: 決済・SAML のモック化により **7%** 台へ。
    *   **中期 (3ヶ月)**: 未実装機能のデリバリーに伴い **4.5%** (5% 以下) を達成・維持する。

## 6. 決済関連を「別枠」扱いするべきかの是非

**結論: E2E スイート内には残すべきだが、実行トリガーと検証レベルを分けるべき。**

*   **妥当性**: PCI-DSS や本番課金リスクを鑑みると、Playwright でクレカ番号をベタ打ちするのは Staging でも推奨されない。
*   **推奨案**:
    *   UI 導線確認 (ボタン表示等) は通常の E2E で実施。
    *   決済実走 (Webhook 含む) は **Stripe CLI を併用した専用のジョブ** として分離。
    *   「スキップ」ではなく「特定環境でのみ実行」という扱いにし、メインのスキップ率メトリクスからは除外する。

## 7. 結論: 実行計画

### 短期 (〜2週間)
*   Stripe CLI をテストコンテナに導入し、R-163, R-171 の Webhook テストを自動化する。
*   プロダクト側に接続元 IP 偽装用の Debug API を追加 (R-139 解消)。

### 中期 (〜1ヶ月)
*   SAML モック IdP (OneLogin モック等) を Staging 環境に接続し、R-109, R-115, R-119 を解消する。
*   プロダクト未実装機能 (R-144, R-166 等) の開発優先度を上げ、テストを有効化する。

### 長期 (継続)
*   スキップ率 5% を「ビルド失敗」の閾値として設定し、無闇なスキップ追加をシステム的に抑制する。
