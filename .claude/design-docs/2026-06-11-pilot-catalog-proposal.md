## 前提メモ

- roadmap が参照する `/tmp/bug-catalog.json` 等は既に消失しており、Slack 実バグ16件との個別相関は roadmap 本文＋ `.claude/product-bugs.md` の記述から間接的に判断した
- 既存実装で**比較的健全**（具体操作＋具体 assertion あり）: auth の AT01/AT02（ログイン・PW変更）、auth-180（権限拒否）、auth-260（テナント分離）、auth-270（Cookie属性）、records の RC03〜RC13（一括編集モーダル・一括削除件数減・保存永続化・編集ロック）

## 2. 不採用の代表例と理由（11件）

| # | 元ケース | 内容 | 不採用理由 |
|---|---|---|---|
| 1 | auth-070〜100 (旧212-1〜4) | 同時ログインプラン（人数制限・強制ログアウト・60分自動ログアウト・全端末ログアウト） | 60分自動ログアウトは実時間待ちが必要で E2E 不適。強制/全端末ログアウトは既存 spec で「他セッション即時無効化が製品未実装」として skip 実績（auth-150/170）。製品仕様確定後に再検討 |
| 2 | auth-110〜140 | 2段階認証（設定・QR・TOTP成功/失敗） | trial env の admin ID が非メール形式で 2FA UI 自体が描画されない製品制約（auth-120/130 で skip 実績）。メール形式 ID のデータセットアップ整備が先 |
| 3 | auth-240/245 (旧B003) | 20回失敗でアカウントロック | trial env は lock_timeout_min=0 でロック機能無効（skip 実績）。かつ20回ログイン試行は実行時間が長い。設定有効化込みフロー設計＋環境整備後に追加 |
| 4 | auth-250 / auth-290 | SAML 実ログイン / mTLS クライアント証明書 | 外部 IdP・ALB mTLS 経路依存で E2E 再現不可（test-env-limitations 該当）。「設定画面が開く」だけの確認はパイロットの価値基準（壊れたら業務が止まる）を満たさない |
| 5 | auth-280 | InternalAuthMiddleware（X-Manage-Key）認証 | VPN/内部管理キーのインフラ認証で、ユーザージャーニーではない。PHPUnit/Integration 側の責務 |
| 6 | auth-010の一部 (旧144-01) | 推奨ブラウザ以外で警告バー表示 | 警告表示の細部で、壊れても業務は止まらない（P3相当）。パイロット枠から除外 |
| 7 | rec-010 (旧143-01) | 一覧コメントアイコンのマウスオーバーで件数・最新投稿時間表示 | 表示の細部。コメント機能の本線は comments-logs 領域のカタログで扱うべき |
| 8 | rec-070 (旧237) | 一覧上部スクロールバー操作 | 既存実装でも「スクロール可能要素が見つからなければ素通り」の曖昧テスト。観測可能な業務的価値が薄い |
| 9 | rec-080〜100 (旧35-1/52-1/52-2) | 関連レコード一覧フィールドの必須未入力バリデーション・参照中テーブル削除エラー | フィールド設定（table-definition）領域のシナリオでレコードCRUDジャーニーではない。バリデーション網羅は PHPUnit 側の責務。参照整合性（旧35-1）は table-definition カタログで採用検討 |
| 10 | rec-110〜140 (旧281/322/502/518) | 過去の個別バグ修正確認（特定顧客テーブル「契約書管理簿」「T_入金_HEAD」依存） | 実環境の特定テーブル前提で使い捨てテスト環境では再現不可。一般化可能な観点（数値小数の編集画面表示=旧322）は fields 領域カタログへ |
| 11 | rec-040/050 (旧180-2/180-3) | 権限なしデータ混在時の一括編集拒否・ロック中データの強制上書き | 複数ユーザー＋権限グループ＋ロック状態の組み合わせセットアップが重く、既存 spec でも skip 実績。権限制御の本線は users-permissions 領域カタログへ。ロック注意書きの表示確認は rec-009 に統合済み |

## 3. 判断に迷ったもの（ユーザー判断が必要）

1. **auth-190（閲覧のみユーザーのAPI直接更新拒否）を採用すべきか**
   - セキュリティ的には P1 級の観点だが、`.claude/product-bugs.md` に「trial env で権限制御 API が機能しない疑い（一般ユーザーが master admin API 更新で 200）」が記録済みで、採用すると当面 fail/skip が常態化する
   - 案A: 採用して fail のまま残す（プロダクトバグ可視化）／案B: 製品側の調査決着までカタログ外で product-bugs 管理 → **どちらにするかご判断ください**

2. **auth-008（マルチテナント分離）の実行コスト**
   - テナントを追加で1つ作成するため create-trial を2回叩く（1シナリオで数分＋staging RDS にDBが1個増える）。セキュリティ価値は高いが、パイロットの「毎回実行する」運用に乗せるか、週次等の低頻度枠にするか

3. **編集ロックの「他ユーザー視点」検証（rec-010 の拡張)**
   - 現行案は単一ユーザーのロック取得/解除ライフサイクルのみ。ロックの本質（ユーザーAが編集中はユーザーBが編集できない）は2ブラウザコンテキストが必要で、過去 spec では skip されてきた領域。パイロットで踏み込むか、第2弾に回すか

4. **「ソート」シナリオの不在**
   - 厳選基準に「ソート/フィルタ」が挙がっているが、auth/records の yaml にソートの明示ケースが存在しない（layout-ui/uncategorized 側にも明確なものを発見できず）。一覧カラムのソートを**新規シナリオとして起こす**か、ソース無しのためパイロット対象外とするか

5. **auth-005（パスワードポリシー）の destructive リスク**
   - バリデーションエラー確認のみの設計だが、操作対象がマスター自身のパスワード編集画面のため、製品バグや誤操作で実際にパスワードが変わると環境が壊れる。destructive: true とした上で採用したが、テストユーザーのパスワード編集画面に付け替える設計（マスターを触らない）も可能。どちらを取るか

6. **旧322（数値小数 1.00 が編集画面で 1 になる）の扱い**
