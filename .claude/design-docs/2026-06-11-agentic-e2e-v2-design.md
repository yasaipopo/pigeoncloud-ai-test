# エージェント実行型 E2E v2 設計書（ハイブリッド方式）

作成日: 2026-06-11
ステータス: ユーザー承認済み（方式・環境使い回し含む）。パイロット実走前。

## 背景・課題

- 現行: 36 spec / 約898テスト / yaml数千ケース。全完走に半日以上、かつ gemcli レビュー（2026-05-09）で **434テスト中77%が違反**（navbar+ISE不在のみの偽装テスト、assertion緩和）
- 構造的問題: ①ケース数過多で完走不能 ②pass率を上げる圧力でテストが空洞化（skip化・耐性化の連続コミット） ③セレクタ保守地獄
- ロジック網羅は PHPUnit/Integration 側の責務とし、E2E は「フロントから主要機能をかいつまんで通す」役割に再定義する
  - **旧方針「E2Eでも全パターン網羅」（CLAUDE.md 2026-04-20改訂）は廃止**（2026-06-11 ユーザー指示）

## ゴール

- 厳選した 150〜300 シナリオが **100% 本物の検証で** 完走する状態
- 運用: リリース前スポット実行（一晩回しっぱなしOK・途中スタックしないことが最優先）

## 方式: シナリオカタログ + エージェント実行 + 検証分離ハイブリッド

### 1. シナリオカタログ（唯一の正）

```
catalog/
├── auth.yaml
├── records.yaml
└── ...（領域ごとに1ファイル）
```

形式:

```yaml
- id: auth-001
  title: マスターユーザーでログイン・ログアウトできる
  priority: P1            # P1=コア動線 / P2=主要機能 / P3=余裕があれば
  destructive: false      # true = 環境状態を壊す系（後述の隔離対象）
  precondition: 自分専用のリソースをこのシナリオ内で作成
  steps:                  # 自然言語。セレクタは書かない（エージェントが画面を見て解決）
    - /admin/login でID・パスワードを入力しログイン
    - 右上メニューからログアウト
  observations:           # PASS判定の証拠物。「正常に〜」等の曖昧表現は禁止
    - ログイン後のURLが /admin/dashboard でナビゲーションバーが表示されている [スクショ]
    - ログアウト後 /admin/login に戻り、/admin/dashboard 直アクセスでログインへリダイレクトされる [スクショ]
```

ルール:
- **observations は観測可能な具体値**で書く（URL・表示文字列・件数・色など）。曖昧な期待結果はカタログレビューで弾く
- **グローバル状態依存の禁止**: 「テーブル数がN件」等は不可。「自分が作成したテーブル X が一覧に表示される」のように自リソース基準で書く
- 厳選基準: ①コアユーザージャーニー ②Slack実バグとの相関（`.claude/e2e-coverage-roadmap.md` の分析流用） ③Integrationで見切れないUI統合経路
- **厳選リスト（何を採り何を捨てるか）はユーザーレビュー必須**

### 2. 実行アーキテクチャ（1スポット実行）

```
オーケストレーター
  ① テスト環境を直列で N 個作成（create-trial競合回避）→ 1エージェント1環境を割当
  ② 実行エージェント (Sonnet) × N 並列、各自の環境内でシナリオを直列消化
       キャッシュスクリプトあり → 再生（安価・数十秒）
       なし/壊れた → 使い捨て Playwright スクリプトを書いて実行→観察→修正のループで完遂
       （MCP Playwright は使わない。CLIスクリプト方式）
       ハードリミット: 1シナリオあたり修正試行 最大3回。超過は STUCK_RETRY_EXCEEDED として
       記録しスキップ（コスト暴走防止。翌朝人間がカタログ/難易度を調整）
       成果物: evidence/{id}/ にスクショ + 観測値JSON + 実行スクリプト
       スクショには実行ID（タイムスタンプ+シナリオID）を DOM 注入で画面隅に写し込む（証拠の真正性担保）
  ③ 判定エージェント（実行と別エージェント。作成・評価分離ルール準拠）
       observations と証拠物を突き合わせ PASS / FAIL / EVIDENCE_NG の三値判定
       「観測値が証拠に写っていないPASS」は EVIDENCE_NG として差し戻し
       スクショ内の埋め込み実行IDの一致も確認（古い/他テストの証拠流用を排除）
       EVIDENCE_NG 時はオーケストレーターが「追加証拠の撮影指示」付きで1回だけ自動再実行。
       それでもNGなら EVIDENCE_NG のまま朝レポートへ + カタログ observations の改善提案を併記
  ④ 成功スクリプトを cache/scripts/{id}.js に保存
  ⑤ results.json 集約 → sheet.html 同期 → サマリーレポート
```

- **三値判定が偽装PASS再発の構造的防止**: テストの甘さを FAIL と区別して検出する
- FAIL 時の成果物（再現スクリプト + スクショ + 失敗画面）はそのまま `.claude/product-bugs.md` とバグ報告に使う

### 3. テスト環境のライフサイクル（1エージェント1環境・使い回し）

2026-06-11 ユーザー指示: 「テストごとに環境を毎回作る」のをやめ、**1エージェント1環境を作って実行内で使い回す**。

- 旧来のリセット理由（メニュー数・テーブル数変化で決定的スクリプトが壊れる）は、適応型エージェント実行では消滅する
- 効果: create-trial 呼び出しが 36回/実行 → N回/実行（N=並列数 4〜6）。最重量処理の激減 + staging RDS 環境DB蓄積の緩和
- 担保すべき2点:
  1. **データ汚染対策**: 各シナリオは `{シナリオID}-` プレフィックスの自分専用リソースを作成して検証。他シナリオの残骸に影響されない observations 設計（カタログ規約で強制）
  2. **破壊的シナリオの隔離**: システム設定変更・パスワードポリシー・IP制限・ユーザー削除等 `destructive: true` は**各環境の消化順の最後**に回す（環境を壊しても後続被害なし）。ログイン不能級のものは専用使い捨て環境
  3. **グローバル設定の保護**: システム設定（/admin/setting/** 等の全体設定）に触れるシナリオは `scope: global` タグを必須とし destructive 同様に最後尾へ。実行エージェントのプロンプトで「許可されたシナリオ以外でのシステム設定変更は厳禁」を明示（誤操作による連鎖FAIL防止）
- キャッシュスクリプトも使い回し環境前提で書く（自リソース作成から始める・実行ごとにユニーク名）
- 環境は実行単位で使い捨て（実行をまたぐ再利用は将来検討。状態ドリフト蓄積 vs RDS節約のトレードオフ）

### 4. 一晩止まらない仕組み

- シナリオ単位タイムアウト 10分。超過は kill して SKIP 記録、次へ
- `runs/{date}/checkpoint.json` に逐次書き込み。クラッシュしても完了済みスキップで再開可能
- 環境作成失敗・セッション切れ → 環境作り直しリトライ（最大2回）→ 不能ならその担当分を他エージェントへ再配分
- 朝に必ず「完走 or 残件と理由」のレポートを出す

### 5. 既存資産の扱い

| 資産 | 扱い |
|---|---|
| `specs/*.yaml` 数千件 | カタログ厳選の原料。移行後アーカイブ |
| `tests/*.spec.js` 898件 | 健全なものはキャッシュスクリプトの種。偽装テストは廃棄 |
| sheet.html / E2Eビューアー | 継続利用（結果確認UI） |
| パイプライン（check-yaml/spec-create/check-run） | v2で廃止（実行+判定エージェントに置換） |

### 6. パイロット（GO/NO-GO判断用）

1. auth + records の 15〜20 シナリオをカタログ化 → ユーザーレビュー
2. オーケストレーター + 実行 + 判定の最小実装で実走（初回はキャッシュなしフルエージェント）
3. 計測:
   - 1シナリオの時間・トークンコスト（→200件換算で一晩に収まるか）
   - EVIDENCE_NG 検出の有効性（甘い判定を実際に弾けたか）
   - 完走性（スタックなし・STUCK_RETRY_EXCEEDED の発生数）
   - **FAIL 振り分け精度**: FAILのうち本物のプロダクトバグ vs テスト側不備の比率（偽陽性が多いと運用が信頼を失う）
   - 環境使い回し起因の汚染FAILの有無（→リセット頻度の最適値判断）
4. 200件換算の見積もりを提示 → 全面展開はユーザー判断

### 7. カタログの整備・更新プロセス（2026-06-11 追加）

カタログは「少数精鋭を維持する」ことが旧体制（数千件膨張→完走不能）への再発防止。**追加は常にトリアージを通す**。

#### トリガーと3分類トリアージ

バグ修正PR・本番障害・Slack #テスト-staging 報告が出たら、リリース前スポット実行の**前段**でトリアージする:

| 分類 | 判定 | カタログへの反映 |
|---|---|---|
| (a) ロジックバグ | 原因も症状もロジック層。プロダクト側 Unit/Integration で再発防止済み | **反映しない**（E2Eの責務外） |
| (b) UI統合経路のバグ | ユーザーに見えた症状が「動線の断絶」（画面が開けない・保存できない・削除できない等） | **症状の動線**をシナリオ化。バグの内部原因ではなく「ユーザーが通る経路」を書く |
| (c) 既存動線上のバグ | 症状の経路をカバーするシナリオが既にある | 新設せず**既存シナリオの observations に観測点を1行追加** |

原則:
- 追加判断の優先順位は (c) observations追記 > (b) 新設。新設は P1/P2 判定と destructive/scope タグ判定をセットで行う
- プロダクトPRに Unit/Integration テストが含まれている場合、E2E は同じロジックを重複検証しない。**E2E が足すのは「その障害でユーザーが通れなくなった動線」だけ**
- カタログ追加・削除は**ユーザー承認制**（スポット実行レポートに「カタログ更新提案」セクションを設けて採否を仰ぐ）
- 追加時は `source:` に障害ID/PR番号を記録（例: `source: [pigeon_cloud#3593, 本番障害k8n50xe4xy]`）→ 後から「なぜこのシナリオがあるか」を追跡可能に

#### 例: pigeon_cloud#3593（copy-fields 型不正でテーブル詰み）

- 原因（option sanitize ロジック）→ プロダクト側 Unit 19件 + Integration 329行で再発防止済み = (a) として E2E 反映不要
- 症状（AIビルダーで作ったテーブルが開けない・削除できない）→ 「AIテーブルビルダーでテーブル作成 → 一覧が開ける → レコード追加 → テーブル削除」という**動線がカタログ未収載** = (b) として新設候補（AIビルダーが trial 環境で動くか要確認。不可なら test-env-limitations 管理）

#### 新機能リリース時

- リリースに UI 動線の新設・変更が含まれる場合、同じトリアージで「P1動線か？」を判定して追加提案
- カタログ全体の見直しは四半期に1回（使われていない機能のシナリオ降格・P3整理）

#### 自動化（全面展開後の発展）

- merged PR / Slack バグ報告を定期スキャンして「カタログ更新提案」を自動起案するエージェント（旧 roadmap M3 の v2 版）。パイロット段階では手動トリアージで運用

## リスクと対策（要約）

| リスク | 対策 |
|---|---|
| エージェントの甘い自己判定（偽装PASS再発） | 実行・判定の分離 + 証拠物必須 + EVIDENCE_NG 三値判定 |
| 非決定性・再現性なし | FAILは再現スクリプト+trace必須 / 成功はスクリプトキャッシュ化 |
| 使い回し環境のデータ汚染 | 自リソースプレフィックス規約 + グローバル状態依存observations禁止 |
| 破壊的シナリオで環境死 | destructiveタグ + 最後に実行 or 専用環境 |
| 夜間スタック | シナリオ単位タイムアウト + checkpoint再開 + 担当再配分 |
| トークンコスト | 実行はSonnet / 2回目以降はキャッシュ再生でエージェント起動を故障時のみに |
