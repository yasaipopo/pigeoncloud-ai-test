# MCP併用 E2Eテスト設計（v3案）— セットアップはMCP・操作はUI・デグレ再発防止込み

作成: 2026-06-27 / 前提: 基本AIで実行 / 関連: pfc-mcp-api（staging専用・検証バイパスなし・本番無効）, regression-prevention-strategy.md（pigeon_cloud側 2026-06-27）

## 0. 背景・課題（なぜ作り直すか）

- **今**: 準備（テーブル/ユーザー/WF/権限）まで全部 UI クリックでやっており非常に重い（wf-001 が承認WF設定で何度も STUCK したのが典型）。
- **問題を起こすたびに環境・テーブルが増殖**（使い捨て {id}- テーブルが溜まる）。最低限にしたい。
- **デグレ実害**（regression-prevention-strategy.md）: ①仕様変更デグレ（#2885 自己言及表示条件×自動入力×制限権限で下書きが壊れる）②相互干渉（直すと別が再発）③cherry-pickドリフト。「CIは在るテストしか守らない／壊れた組み合わせを誰もテストしていない」のが穴。
- **新ピース**: pfc-mcp-api（MCP）が staging で使えるようになった。テーブル定義・レコード・WF承認・grant を**検証を通る正規フローのまま API で**作れる（UIバイパスではない）。

## 0.5. 🔴 着手前提（独立レビュー C1/C2 反映・2026-06-27）

- **C1 実機ツール確定 → ✅ 解消（2026-06-28）**: pfc-staging MCP を接続し `pfc_staging_ping` で確認した結果、**deploy版=25ツール**（tenant=ai-test/staging）。`pfc_workflow_apply`(WF申請)/`pfc_manage_record`(ユーザー/組織/役職作成)/`pfc_csv_export/import`/`pfc_file_upload/download`/grant更新/field_grants まで MCP 可。よって準備の前倒し範囲は **25ツール版**を正とする（§2 の「UI専用」だった WF申請・ユーザー作成・CSV・ファイルは MCP 準備に寄せられる）。
  - 🔴 **ただし C2 は維持**: MCP でできても、被検証機能（テーブル作成/項目定義/WF設定/通知設定/帳票登録）は **UI生成動線を最低1本残す**（生成動線のUIバグを拾うため）。「準備の高速化に MCP、検証の主操作は UI」は不変。
- **C2 UI生成動線を空洞化させない（規約）**: MCP は `DatasetStoreAction`(サーバ検証)は通るが **Angularフォーム/admin.php controller/クライアント検証/フォーム描画は通らない**（UI=admin.php と MCP=index.mjs は別経路）。準備を全部 MCP に寄せると**生成動線のUIバグが不可視**になり、regression-strategy が指摘する「壊れた種類を誰もテストしていない」穴を再生産する。
  - 🔴 **規約**: 「準備はMCP」は**反復準備の高速化に限定**する。**各主要機能につき最低1本は"UI生成動線"シナリオを必ず残す**（テーブル作成UI／項目追加UI／grant設定UI／WF定義UI／レコード作成UI 等）。これらは action=UI で生成自体を検証し、MCPで代替しない。

## 1. 中核アイデア — 「準備はMCP、検証の主操作はUI」

E2E の価値は「ブラウザの通常動線が動くか」。だが**反復的な準備までUIでやる必要はない**（生成動線そのものの検証は §0.5 C2 で別途残す）。各シナリオを3層に分解し、層ごとに最速の手段を選ぶ。

```
[準備 setup]      → 原則 MCP（テーブル/レコード/WF定義/grant）＋ ユーザーは debug API
                    UI でしか作れない前提物（下書き元データ等）だけ UI
[操作 action]     → UI（Playwright）＝検証の主役。ここは必ずブラウザ通常動線
[検証 verify]     → 画面表示は UI（スクショ＋DOM実値）／ 永続結果は MCP record_list でも裏取り
```

- 「**毎回新規でテーブル作成するテスト**」だけは作成自体が action なので UI で新規作成する（例外）。
- それ以外は**シード済みフィクスチャを使い回す**（§3）。準備の MCP 化で wf-001 型の「設定で詰まる」が消える。

## 2. UI↔MCP 切り分け（実装済み13ツール基準・COVERAGE 2026-06-27）

| 領域 | MCP セットアップ可 | UI 専用（操作 or MCP未対応） |
|---|---|---|
| テーブル定義 | ✅ create/update/delete（全23型・表示/必須条件・unique・calc・select_other_table・relation・settings・WF定義・grant(create時)） | グループ管理・union |
| レコード | ✅ 一括 create/update/delete/list（condition検索・by_label） | **下書き保存**・コメント |
| ワークフロー | ✅ フロー定義・**承認/却下/差戻し**・取り下げ | 🔴 **WF申請（applying開始）はUI専用**（record APIは申請を開始しない） |
| 権限 | ✅ grant(everyone/me/custom)を**create時** | grant の**更新**・field粒度権限・table_editable/author・grant_group作成 |
| ファイル/CSV/帳票 | ⚠️ 帳票DL URLのみ | ファイルUP/DL・CSV入出力・帳票テンプレ定義 |
| 集計/ビュー/ダッシュボード | ❌ | すべて UI |
| ユーザー/組織/役職 | ❌（MCP未対応） | UI ／ ただし**テスト用ユーザーは debug API で作成可** |

→ この表が「**シナリオごとの振り分け**」の根拠。準備に出てくる前提物をこの表で MCP / debug API / UI に割り振る。

## 3. データ方式 — シード済みフィクスチャ使い回し（作りすぎ解消）

環境（レジストリ再利用）の中に、**少数の固定フィクスチャを MCP で一度だけシード**して使い回す。シナリオごとにテーブルを作らない。

| フィクスチャ | 内容（MCPで生成） | 主な用途 |
|---|---|---|
| `fx-basic` | text/number/date/select＋既知の10行 | 検索・ソート・フィルタ・ビュー・集計・CSV・編集 |
| `fx-workflow` | 承認WF有効＋承認ルート（承認者B固定） | WF申請→承認 |
| `fx-perm` | custom grant＋制限ユーザー（debug API） | 権限による表示/操作制限 |
| `fx-schema` | 定義変更検証の専用テーブル | 項目追加・型変更・テーブル設定（変更→検証→**MCPで復元**） |
| `fx-regression` | #2885型（自己言及表示条件×自動入力×制限権限） | デグレ回帰ピン（§5） |

**干渉対策（共有フィクスチャの鉄則）**:
- **フィクスチャは環境ごとに独立 seed**（チェーン内は直列・CLAUDE.md §2）。並列は環境=チェーン単位なので共有フィクスチャの並列競合は起きない（M1）。
- 読み取り系は**行を触らない**（干渉ゼロ）。
- 行変更系は**自分の `{id}-` プレフィックス行だけ**作成・操作し、末尾で**自分の行をMCPで削除**（テーブルは残す＝一覧が汚れない）。
- 定義変更系（H1 対策）: ①変更前に `pfc_get_table` で**スナップショット取得** ②変更→検証 ③**スナップショットへ復元**（手書き逆操作でなくスナップショット適用） ④復元後 `pfc_get_table` で**一致を機械検証**＋ VIEW async rebuild の**反映待ち** ⑤**復元失敗時は当該チェーンを中断し fx-schema を再 seed**（後続への波及防止）。共有が不安なら **fx-schema は使い捨て**（復元不要化）も選択可。
- 🔴 **可視性検証の注意（M2）**: MCP `record_list` は master フォールバックで**全行見える**。だから「制限ユーザーに何が見えるか」の検証は**必ずUI（制限ユーザーのセッション）**で行う。MCP裏取りは**永続性（保存されたか）確認に限定**する。

効果: テーブル/環境の新規作成が激減・一覧にゴミが溜まらない・「既存テーブルを変えて同時チェック」がそのまま定義変更系シナリオになる。

## 4. カタログの「準備プロファイル」拡張

各シナリオに、振り分けを機械可読で持たせる（executor がこの通り準備する）。

```yaml
- id: rec-004
  title: 簡易検索で一覧を絞り込むと一致レコードだけ表示される
  fixture: fx-basic              # 使うフィクスチャ（新規作成しない）
  setup:                         # 準備の手段を明示（MCP/debug/UI）
    - mcp: 既存 fx-basic を使用（行は既知シード）
  action: ui                     # 検証の主操作は必ずUI
  verify: ui                     # 画面表示で判定（必要なら mcp record_list で裏取り）
  cleanup: none                  # 読み取りのみ＝掃除不要
  steps: [...]
  observations: [...]
```

- `fixture` / `setup`(mcp|debug|ui) / `action`(ui) / `verify`(ui|mcp) / `cleanup`(none|rows|restore) を追加。
- バリデータ（validate-catalog.js）に「action は ui 必須」「setup の主操作 UI 代替は禁止」等のルールを足す。

## 5. デグレ再発防止の取り込み（regression-prevention-strategy.md）

E2E（ブラウザ動線）でやるのは**金パス＋壊れやすい組み合わせ**。プロダクト側の Unit/Integration 回帰ピン（L1）とは責務分担しつつ、**ブラウザで見える症状**は E2E 回帰ピンにする。

| デグレ類型 | E2Eで張る回帰ピン（fx-regression 中心） |
|---|---|
| A 仕様変更 | **#2885型**: 自己言及表示条件＋自動入力＋制限権限ユーザーで「下書き保存→再表示でデータ保持＆条件項目が出る」。E2E版は **「ブラウザ下書き保存動線の追加カバー」と位置づける**（H3）。🔴 **反証対照（外すと再現/戻すと消える）は共有stagingでfixトグルを切れないため L1統合テスト の責務**（pigeon_cloud側）。E2Eはピン化前に下記レシピを §に書き切ってから着手: (1)「自動入力」をMCPで何で構成するか（calc項目/default値）(2) 制限ユーザーのUIログイン認証情報の入手（debug APIが既知PWでユーザー作成）(3) grant の hide_fields/only_view_fields と自己言及表示条件の具体構成 |
| B 相互干渉 | 同一フィクスチャで両制約を1シナリオに同居させ、どちらかが壊れたら落ちるようにする（例: 表示条件 と 項目#1550系を同居） |
| C pickドリフト | E2E対象外（プロダクト側スクリプト）。ただし金パスが緑かはリリース前スポット実行でカバー |

→ 「壊れた種類のシナリオをそもそもテストしていない」穴を、**fx-regression に事件シナリオを恒久ピン**として置くことで塞ぐ。新しいデグレが出たら fx-regression に1本追加（カタログ更新トリアージの一種）。

## 6. AI実行パイプラインへの組み込み（既存v2を踏襲）

実行→判定→第三者チェック→トリアージ→レポート は不変。**executor の「準備」フェーズが MCP/debug 主体になる**だけ。
- executor 指示書に「準備は準備プロファイルの setup に従い MCP/debug を使う。action/verify は UI」を追記。
- 第三者チェック（Check ステージ）に「action が UI で行われたか（準備を操作にすり替えていないか）」を追加。
- MCP は staging 専用・本番無効・検証バイパスなしなので安全（準備の近道であって、抜け道ではない）。

## 7. コスト/運用

- フィクスチャ再利用＋MCP準備で**1シナリオの所要時間が大幅減**（wf-001の承認WF設定UIポチポチが消える）。
- 録画: 全件は重いので **既定は失敗時のみ録画**に寄せる選択肢（regression-strategy L4 と一致）。ただしユーザー指示の「全録画」を維持するなら維持（要確認）。
- 環境: レジストリ再利用のまま。フィクスチャは月1でリフレッシュ（肥大時は作り直し）。

## 8. 段階導入

0. 🔴 **Step0（C1）: MCP を本プロジェクトに登録**（.mcp.json＋トークン/テナント）→ `pfc_staging_ping`＋**実装済みツール列挙**で deploy版（13か25か）を確定 → §2 切り分け表をその版にピン留め。
   - 🔴 **テナント allowlist（H2）**: registry の各環境は別テナント(DB)。MCP は `X-Mcp-Tenant` がサーバ allowlist に無いと不可。**provision-envs / レジストリ更新時に新規テナントを MCP allowlist へ自動登録**する手順を組み込む（これが無いと作り直した環境でMCP準備が即詰まる）。allowlist 運用主体・master adminid も確定。
1. フィクスチャ seed スクリプト（MCP）を作る（fx-basic/workflow/perm/schema/regression）。**環境ごとに独立 seed**。
2. 既存21シナリオの precondition を「準備プロファイル」に移行（使い捨てテーブル→フィクスチャ）。**各主要機能の"UI生成動線"シナリオ（§0.5 C2）は MCP化せず残す**。
3. #2885型 回帰ピン（fx-regression）を1本追加＝**ブラウザ下書き動線カバー**として（反証対照は L1 の責務・H3 のレシピを書き切ってから）。
4. マニュアル準拠の網羅を、準備プロファイル方式で第1バッチから展開。
