1. 何をテストするか（スコープ）

E2E は「フロントから主要動線をかいつまんで、ブラウザの通常ユーザー操作で通す」役割に限定する。ロジックの網羅・攻撃可否・エッジは pigeon_cloud 側の Unit/Integration の責務。

区分E2E でやる 対象E2E でやらない 範囲外
観点マニュアルどおりに基本動作が動くか（通常動線）認可バイパス・攻撃・並行性・型不正・境界
操作ブラウザUI操作。テスト値はブラウザ表示から取得raw API 直叩きで検証の主操作を代替
例（承認WF #3696）申請→承認→承認済みが通るか第三者が攻撃で取り下げできるか（=integration test）

※ データ準備（debug API でのテーブル/ユーザー作成）は可。禁止は「検証の主操作を API で代替」すること。

2. シナリオの源泉 — 公式マニュアルが「正」

期待値（observations）は 公式ヘルプ/マニュアル（pigeon-help-2026）の記載が根拠。Claude の推測仕様は使わない。バグ修正PR・障害が来たら3分類でトリアージし、カタログ（唯一の正＝catalog/*.yaml）を少数精鋭で更新する（ユーザー承認制）。

図1: マニュアル準拠 + バグトリアージ → カタログ（SSoT）への流れ。spec.js は廃止し、テストコードはエージェントが生成する使い捨て/キャッシュ。

3. 実行フロー（中核）— 1回のスポット実行

環境準備のあとは保存版ワークフロー e2e-v2-spot-run が全自動で回す。各シナリオは 実行 → 判定 → 第三者チェック → (FAIL/STUCKなら)トリアージ → レポート/ビューアー を通る。

図2: スポット実行1回の中核フロー。①〜⑥のうち②〜⑥は全自動。実行と判定は別エージェント（自己採点しない）。

4. 偽装PASS防止の5重ガード

旧体制の死因は「pass数を稼ぐためのテスト緩和（77%が偽装テスト）」。v2 は緩和を構造的に潰す。

#ガード防ぐもの
1実行と判定のエージェント分離自己採点による甘い合格
2証拠主義（観測ごとにスクショ+DOM実値）「やった気」報告
3スクショ右下の実行IDバッジ古い/他テストの証拠流用
4三値判定（PASS/FAIL/EVIDENCE_NG）証拠不十分の取りこぼし
5第三者チェック（要件すり替え検出）要件の簡略化（例: 別人承認を同一userで代替）実害から追加

実証: wf-001（承認WF）で executor が「申請者≠承認者」を master 自己承認に簡略化して偽PASS→ガード5が検出→厳密化後は別人承認で正規PASS。

5. 環境モデル — 1チェーン1環境・使い回し・月次保管

図3: 環境はリセットせず使い回す（適応型エージェントなのでメニュー数変化で壊れない）。各シナリオは {id}- プレフィックスの自リソースのみ作成。destructive/global は最後尾。

成果物（report.md / evidence: png+json / video: webm+mp4）は月ごとに集約。trial上限対策に末尾でプレフィックスリソースを掃除。

6. 結果の見方 — 自己完結ビューアー

node v2/lib/build-viewer.js がアーカイブを走査して ~/pigeon-e2e-archive/index.html を生成（サーバ不要・開くだけ）。実行（月別）→シナリオ（status/triage）→観測値・スクショ・動画を1画面で。ワークフローが実行末尾で自動再生成する。旧 spec/Sheetベース viewer は廃止

7. 役割分担

役割担当やること / やらないこと
オーケストレーターClaude メイン環境準備・起動・集計・レポート提示。ブラウザ操作も判定もしない
実行エージェントSonnetシナリオ完遂と証拠出力のみ。判定しない。要件を緩めない
判定エージェントSonnet（別文脈）証拠だけで三値判定。実行側の主張を信用しない
第三者チェック/トリアージ別エージェント/gemcli要件すり替え検出・FAIL/STUCK分類
ユーザー（石川さん）—カタログ採否・バグ報告判断・GO/NO-GO
