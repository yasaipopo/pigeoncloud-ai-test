0. ゴール定義（完了の状態）

ゴール: 公式マニュアル準拠の厳選 E2E カタログが、MCP準備＋UI操作で 100% 本物の検証で完走する。 偽装PASSは構造的に不可。過去デグレは回帰ピンで再発検知。結果はビューアーで一覧。新規テストはトリアージ→承認→カタログ化の定常運用に乗る。全工程 AI 実行。
完了条件判定
カタログが機能エリアを網羅（マニュアル準拠・少数精鋭）主要エリアに P1 シナリオが存在
各シナリオが準備プロファイル（MCP/debug/UI）で動くvalidate-catalog 緑＋実走 PASS
UI生成動線シナリオが各機能に最低1本（C2）カバレッジ表で確認
デグレ回帰ピン（#2885型 ほか）が存在fx-regression に登録・実走 PASS
結果がビューアーで月次閲覧できるindex.html 自動再生成

2. これまでのテストの扱い・移行

資産現状扱い
旧 spec.js（約4,700／v1）CI外・6.5h・77%偽装テスト廃止 1:1移行しない。厳選カタログが置換。tests/ はアーカイブ参照のみ
v2 カタログ 21件（auth8/records12/wf1）使い捨て{id}-テーブル準備移行 準備をフィクスチャ＋MCPに置換（下記手順）
結果ビューアー / アーカイブ稼働中継続

21件の移行手順（1件ずつ）

  • カタログの該当シナリオに 準備プロファイル（fixture / setup / action / verify / cleanup）を付与
  • precondition の「{id}-テーブル新規作成」を「fx-* を使う（行は {id}- プレフィックス）」に書き換え
  • 準備の主操作を MCP/debug に移す（action=UI は維持＝検証の主操作は触らない）
  • UI生成動線シナリオ（C2）は移行対象外で残す（rec-001 レコード作成UI 等は UI 準備のまま＝生成動線を検証）
  • 1件移行ごとに実走 → PASS（＋第三者チェック通過）を確認 → ビューアーで証跡確認

4. チェック方法（どう検証するか）

各シナリオは下記パイプラインを通る。実行と判定は別エージェント（自己採点しない）。PASSでも第三者チェックを通す。

図2: チェックパイプライン。観測ごとにスクショ＋DOM実値、可視性はUI、永続性はMCP record_list で裏取り。FAIL/STUCKは自動トリアージ。

チェックの観点（何を確認するか）

観点方法
画面表示（動線が動くか）UIスクショ＋DOM実値。判定エージェントが画像を実際に開いて位置・内容確認
永続性（保存されたか）MCP record_list / get_table で裏取り
可視性（制限ユーザーに何が見えるか）必ずUI（制限ユーザーのセッション）。MCPはmasterで全部見えるので不可
要件すり替えが無いか第三者チェックが「シナリオ要件 vs 実操作手順」を突合（別人性・UI操作性・準備のすり替え）
マニュアルと一致するかobservations の期待値はマニュアル記載が根拠。判定もそれに照らす

5. 新しいテストへの対応（定常運用）

バグ修正PR・本番障害・Slack報告・新機能が来たら、3分類トリアージ→カタログ更新（ユーザー承認制）。

図3: 新規対応フロー。(b)(c) は source: にPR/障害ID記録→ユーザー採否→カタログ化。デグレは fx-regression に回帰ピン追加。
  • 新シナリオには 準備プロファイル（fixture/setup/action/verify/cleanup）を必ず付ける
  • 複数ロール動線は observations に別人性を必ず入れる（wf-001 教訓）
  • マニュアルに期待値が無い挙動は推測で書かず「要マニュアル確認」
  • 追加は少数精鋭・ユーザー承認制（カタログ膨張を防ぐ）

6. フェーズ別タスク（詳細チェックリスト）

P0 前提確定 最優先

  • pfc MCP を .mcp.json に登録（トークン/テナント）
  • pfc_staging_ping＋ツール列挙で deploy版（13 or 25ツール）を実機確定（C1）
  • 確定版で §UI↔MCP切り分け表をピン留め（WF申請/ユーザー作成/CSV/ファイルの可否）
  • 未決3点（録画 全/失敗時・フィクスチャ粒度・MCP登録方法）をユーザー決定

P1 基盤

  • フィクスチャ seed スクリプト（MCP）: fx-basic/workflow/perm/schema/regression を環境ごとに seed
  • テナント allowlist 自動登録を provision-envs に組込（H2）
  • カタログに準備プロファイル schema 追加＋ validate-catalog にルール（action=ui必須 等）
  • executor 指示書に「準備=MCP/debug、操作=UI」を明記＋ MCP 呼び出し手順
  • 第三者チェックに「準備を操作にすり替えていないか」を追加

P2 既存移行 / P3 漏れ補完 / P4 回帰ピン

  • P2: 21件を1件ずつ移行→実走PASS（§2手順）
  • P3: マニュアル準拠 P1 Top5（tbl/fld/csv-imp/agg/perm）→ P2エリアへバッチ展開。各機能に UI生成動線1本（C2）
  • P4: #2885型回帰ピンのレシピ確定（自動入力構成・制限ユーザー認証・grant/条件）→ fx-regression に実装→実走

7. 運用（P5 定常）

  • リリース前に スポット実行（e2e-v2-spot-run）。環境はレジストリ再利用＋フィクスチャ使い回し
  • 結果は ~/pigeon-e2e-archive/index.html で月次閲覧（自動再生成）
  • バグ/障害/新機能 → 3分類トリアージ → カタログ更新提案 → ユーザー採否
  • デグレが出たら fx-regression に回帰ピン1本追加（恒久検知）
  • フィクスチャは肥大時に月次リフレッシュ（再 seed）
