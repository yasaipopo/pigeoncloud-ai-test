0. 課題 — なぜ作り直すか

課題中身
準備が重いテーブル/ユーザー/WF/権限まで全部UIクリックで用意（wf-001がWF設定UIで何度もSTUCK）
作りすぎ問題のたびに環境・使い捨てテーブルが増殖。最低限にしたい
デグレ実害 A/B/C仕様変更(#2885) / 相互干渉 / pickドリフト。「CIは在るテストしか守らない＝壊れた組合せを誰もテストしていない」
新ピース MCPpfc-mcp-api が staging で使える。検証を通る正規フローのままテーブル定義/レコード/WF承認/grantをAPIで（本番無効・UIバイパスではない）

1. 中核 — シナリオを3層に分解

E2Eの価値は「ブラウザの通常動線が動くか」。だが反復的な準備までUIでやる必要はない。層ごとに最速手段を選ぶ。

図1: 準備=MCPで高速化、操作=UIで通常動線を本物検証、検証=画面UI＋MCP裏取り。「毎回新規作成のテスト」だけ作成自体がactionなのでUI新規作成（例外）。

2. UI↔MCP 切り分け 実装版は要実機確認

これが「シナリオごとの振り分け」の根拠。下表は保守的な 13ツール版（COVERAGE）基準。25ツール版が live なら WF申請・ユーザー作成・CSV/ファイルも MCP に寄せられる（§6 C1）。

領域MCPセットアップ可 準備を高速化UI専用 操作 or MCP未対応
テーブル定義create/update/delete（全23型・表示/必須条件・unique・calc・参照・WF定義・grant(create時)）グループ管理・union
レコード一括 create/update/delete/list（検索・by_label）下書き保存・コメント
ワークフローフロー定義・承認/却下/差戻し・取り下げWF申請(applying開始)
権限grant(everyone/me/custom)をcreate時grant更新・field粒度・table_editable/author・group作成
ファイル/CSV/帳票帳票DL URLのみファイルUP/DL・CSV入出力・帳票テンプレ定義
集計/ビュー/ダッシュ—すべてUI
ユーザー/組織/役職—（MCP未対応）UI ／ テスト用は debug API で作成

3. フィクスチャ — シード使い回しで作りすぎ解消

環境（レジストリ再利用）内に、少数の固定フィクスチャをMCPで一度だけ seedして使い回す。シナリオごとにテーブルを作らない。

図2: テーブル/環境の新規作成が激減・一覧が汚れない。「既存テーブルを変えて同時チェック」＝fx-schema の定義変更系シナリオ。

M2: MCP record_list は master で全行見える＝可視性検証はUI(制限ユーザー)で。MCP裏取りは永続性確認に限定。

4. カタログの「準備プロファイル」

各シナリオに振り分けを機械可読で持たせ、executor がこの通り準備する。

- id: rec-004
  fixture: fx-basic          # 使うフィクスチャ（新規作成しない）
  setup: [mcp]               # 準備の手段（mcp / debug / ui）
  action: ui                 # 検証の主操作は必ずUI
  verify: ui                 # 画面で判定（必要なら mcp record_list で裏取り）
  cleanup: none              # none / rows / restore

validate-catalog に「action=ui 必須」「準備の主操作をUI代替しない」等を追加。

5. デグレ再発防止の取り込み

E2E は金パス＋壊れやすい組合せを担う。fx-regression に事件シナリオを恒久ピンとして置く。

類型E2Eで張る回帰ピン
A 仕様変更#2885型（自己言及表示条件×自動入力×制限権限で「下書き保存→再表示で保持＆条件項目が出る」）をブラウザ下書き動線カバーとして。反証対照はL1統合テスト(プロダクト側)の責務。ピン化前に「自動入力の構成・制限ユーザーのログイン認証・grant/条件の具体」をレシピ化
B 相互干渉同一フィクスチャに両制約を同居＋observationsを制約別に分け、どちらが落ちたか特定可能に
C pickドリフトE2E対象外（プロダクト側スクリプト）。金パスが緑かはスポット実行でカバー

6. 着手前提 — 独立レビューの Critical/High

C1 一次資料が矛盾（最優先） — COVERAGE=13ツール vs AI-REFERENCE=25ツールで、WF申請・ユーザー作成・CSV/ファイルの可否が割れる。着手前に pfc_staging_ping＋ツール列挙で deploy版を実機確認し、§2 をその版にピン留め。
C2 UI生成動線の空洞化防止 — MCPはサーバ検証は通るが Angularフォーム/admin.php/描画は別経路。各主要機能につき最低1本は"UI生成動線"シナリオを必ず残す（テーブル作成UI/項目追加UI/grant設定UI/WF定義UI）。MCP化は反復準備の高速化に限定。
High対策
H1 fx-schema 復元失敗で後続破壊get_tableスナップショット復元＋一致機械検証＋async待ち＋失敗時re-seed（or 使い捨て化）
H2 動的trial環境のMCPテナント未登録provision時に新規テナントをMCP allowlistへ自動登録（無いと作り直した環境で即詰まる）
H3 #2885レシピ不完全E2Eは「下書き動線カバー」と位置づけ・反証対照はL1・欠落レシピを書き切ってから着手

総合評価: 条件付きGO（方向性は妥当・C1/C2/H1〜H3を解消してから着手）。

7. 段階導入
