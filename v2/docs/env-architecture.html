2. 環境のライフサイクル 現状

① 実行開始時に作成

  • create-trial API でこのスポット実行専用のテナントを2個、直列に新規作成
  • 旧体制は「specごとに毎回作成」（36回/実行）→ 今は2回/実行
→

② 実行中: リセットなし・使い回し

  • reset-all 等のリセットは一切呼ばない
  • 各シナリオは rec-008- のような自分専用プレフィックスのテーブル/ユーザーを作って検証（他の残骸と干渉しない）
  • 環境を壊しうる destructive シナリオ（PW変更強制等）は各チェーンの最後尾
→

③ 実行後: 使い捨て（現状）

  • 削除せず放置（従来運用と同じ）
  • 次回スポット実行ではまた新規作成 ← ここが見直し対象

「メニュー数が変わると壊れるからリセット」していた旧体制の前提は、画面を見て適応するエージェント実行では不要になったため、実行内のリセットは全廃しました。

3. 提案: 実行をまたいだ環境再利用 次回から

ご指摘どおり「一度作った環境の再利用」を既定にするのが合理的です。create-trial は最重量処理（1回数分・504リスク・staging RDS蓄積）で、環境作成自体はテスト対象ではなく単なるセットアップだからです。「作る経路のエラーは影響範囲が小さい」という判断にも同意です（壊れても気づくのはセットアップ時で、ユーザーデータには波及しない）。

環境レジストリ方式（提案）

  • v2/envs-registry.json に作成済み環境を永続記録
  • 実行開始時: レジストリの環境にヘルスチェック（ログイン可? 上限余裕あり?）→ OKなら再利用
  • NG（ログイン不能・上限到達・消滅）の環境だけ作り直し（自動回復）

再利用の管理ポイント（対策込み）

  • trialプラン上限: 今回も max_user=5 に到達。→ 実行末尾にシナリオプレフィックスのテーブル/ユーザーを掃除（cleanup ステップ）
  • destructive の後遺症(PW変更等): destructive 専用環境を分離 or 実行後にその環境だけ作り直し
  • 状態の肥大でUIが重くなる: ヘルスチェックで閾値超えたら作り直し（月1程度の自然リフレッシュ）
毎回新規作成（現状）レジストリ再利用（提案）
実行開始までの時間環境数 × 2〜4分ヘルスチェックのみ（数十秒）
staging RDS 蓄積毎回 N 個のDB追加追加ほぼゼロ（壊れた時だけ）
create-trial 504リスク毎回さらされる初回と故障時のみ
リスクなし（常にクリーン）上限・後遺症 → 上記対策で吸収

4. 役割の整理（誰がどの環境を触るか）

主体環境との関わり
オーケストレーター（メインセッション）環境の作成/ヘルスチェック/割当のみ。ブラウザ操作はしない
実行エージェント（Sonnet）割当てられた1環境のみ。自分のプレフィックス外のデータ変更・システム設定変更は禁止
判定エージェント（Sonnet）環境に触らない（証拠ファイルだけ見る）
